description: "Start with one focused task and learn the review loop."

Your first Neroli session should feel small and controlled. Pick one task where
you can tell whether the result is right: a visible bug, a failing check, a
small feature, a test, or a documentation change.

Neroli will give that task its own branch, keep the agent's panes attached to
the work, and bring you back to a review decision instead of leaving you with a
messy checkout.

## Pick a reviewable task

Good first tasks:

- Fix one visible UI issue.
- Add one focused test.
- Update one documentation page.
- Reproduce one reported bug.
- Try one implementation approach.

Avoid broad prompts like "clean up the app" or "improve the frontend." Those
are projects, not reviewable sessions.

## The first loop

<Steps>
  <Step title="Open the repo">
    Select the Git repository in Neroli. This becomes the workspace for your sessions.
  </Step>

  <Step title="Start a new session">
    Create a session for the task. Neroli keeps it from colliding with your main checkout or other sessions.
  </Step>

  <Step title="Give the agent a clear job">
    Say what should change, where to look first, and what should stay untouched.
  </Step>

  <Step title="Watch for evidence">
    Use the session view to follow the plan, progress, output, changed files, and attached panes.
  </Step>

  <Step title="Review the proposal">
    When the session finishes, inspect the diff and verification before you merge, continue, or archive it.
  </Step>
</Steps>

## A good first prompt

```txt
Fix the empty-state layout in the sidebar.

Use the existing design tokens. Do not change backend code.
After the fix, run the check that covers this area and summarize what changed.
```

Add context when it helps:

- A screenshot of the broken state.
- The route, screen, or command that reproduces the issue.
- Files the agent should inspect first.
- Constraints such as "do not change the API" or "keep the existing design system."

## What to trust

Trust the evidence, not the completion message:

- The diff matches the task.
- The plan or summary matches the actual changes.
- The commit story explains the user-facing change instead of hiding behind a wall of diff.
- The app, test, browser flow, or simulator state was verified from this session.
- No unrelated files, local secrets, or generated noise were included.
