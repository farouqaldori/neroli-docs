A good Neroli workspace is predictable. Every session should know how to install
what it needs, start the app, verify changes, and clean up after itself.

Use repo-owned actions for anything the team repeats. They appear in the app so
you and your agents can run the same workflow from the right session.

## Decide what should be repeatable

Start with the workflows that remove guesswork:

| Action | Use it for |
| --- | --- |
| Set up session | Install dependencies, copy local config, prepare generated files. |
| Run | Start the app, server, watcher, or local stack. |
| Build | Compile the main project. |
| Verify | Run focused checks before review. |
| Clean up | Stop local processes or remove generated state. |

## Store actions in the repo

Each action lives in its own directory:

```txt
.neroli/
  automations/
    run/
      automation.json
      run.sh
    verify/
      automation.json
      run.sh
```

`automation.json` describes how the action appears and when it runs. `run.sh` performs the work.

## Example

```json .neroli/automations/verify/automation.json
{
  "title": "Verify",
  "icon": "checkmark.circle",
  "runner": "shell",
  "trigger": { "kind": "manual" },
  "launchers": [
    { "kind": "topBar", "pinned": false }
  ],
  "order": 2
}
```

```sh .neroli/automations/verify/run.sh
#!/usr/bin/env bash
set -euo pipefail

cd "$NEROLI_WORKTREE_PATH"
npm run check
```

<Tip>
  Keep action scripts non-interactive and session-safe. They should not assume your main repository path.
</Tip>

## Local-only setup

Do not commit secrets. Use your setup script to copy or generate local files when a session needs them.

Common examples:

- `.env.local`
- local certificates
- generated config
- seeded development databases
