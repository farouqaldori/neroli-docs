Parallel sessions are useful when each stream can be reviewed independently.
Neroli gives each stream its own lane so you can compare, continue, merge, or
discard without mixing state.

The important decision is simple: should this work share one branch, or should
it move independently?

## Choose the right shape

| Use this | When |
| --- | --- |
| One session | The task is small and one agent can finish it end to end. |
| Multiple agents in one session | The work should share one branch, such as implement plus review or fix plus test. |
| Multiple sessions | Tasks should move independently or become separate pull requests. |
| Competing sessions | You want to compare different approaches to the same problem. |

## Good parallel tasks

- One session updates documentation while another fixes a UI bug.
- One session implements a feature while another explores a refactor.
- One session writes tests for known behavior while another works on implementation.
- Two sessions try different designs for the same problem.

## Poor parallel tasks

- Two sessions editing the same files when you already know both changes must merge together.
- A broad prompt like "clean up the app" with no boundaries.
- Work that depends on shared local state that cannot run in separate directories.

## Review parallel work

Review each branch on its own first. Then decide whether to merge one, merge
several in sequence, or combine work manually.

<Warning>
  Parallel work increases review load. Use it when independence is real, not just because more agents are available.
</Warning>
