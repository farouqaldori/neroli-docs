Verification is how a session earns trust. Run checks from the same session that
owns the changes, then keep the result with the review.

## Testing paths

<Tabs>
  <Tab title="Saved action">
    Use `Verify`, `Build`, or a project-specific action when the check should be repeatable and visible in Neroli.
  </Tab>

  <Tab title="Terminal">
    Use a terminal pane for ad hoc commands, debugging, or one-off checks.
  </Tab>

  <Tab title="Browser">
    Use a browser pane when the change needs visual or interaction verification.
  </Tab>

  <Tab title="iOS simulator">
    Use the simulator pane when a mobile change needs screenshots, touch input, or accessibility inspection.
  </Tab>
</Tabs>

## Recommended checks

| Project type | Example check |
| --- | --- |
| Swift macOS app | `cd macos && scripts/build.sh` |
| Backend | `cd backend && npm run check` |
| Website | `cd website && npm run check` |
| Docs | Mintlify validation or local docs preview |

## What to record

Ask the session to report:

- Which commands it ran.
- Whether they passed.
- Any warnings or skipped checks.
- What still needs human review.

<Info>
  Passing tests are not a substitute for diff review. They only lower the risk of accepting the change.
</Info>
