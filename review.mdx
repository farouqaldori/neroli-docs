Neroli lets you run a lot of work at once. Review is where you decide what lands
back in the main branch.

## One session

<Steps>
  <Step title="Scan the story">
    Start with the plain-language summary of what changed and why.
  </Step>

  <Step title="Inspect the diff">
    Look for unrelated files, generated noise, secret files, dependency changes, and scripts you did not ask for.
  </Step>

  <Step title="Run the workspace checks">
    Use the same Build, Verify, browser, simulator, or terminal automations the workspace exposes.
  </Step>

  <Step title="Decide">
    Merge, open a pull request, continue the session, manually edit, or archive it.
  </Step>
</Steps>

## Many sessions

Parallel sessions are where Neroli opens up. You can let several agents explore,
then pick the work that should continue.

Common patterns:

- one session builds a feature while another updates docs
- several sessions explore different implementations
- one agent writes tests while another changes the product
- one cloud session keeps working while you continue locally

Review each branch on its own first. Then merge one, merge several in sequence,
or combine work manually when that is the right call.

## Red flags

- the diff is larger than expected
- tests were deleted instead of fixed
- verification ran from another session
- cloud returned changes that were not checked locally
- conflict warnings show another session touched the same file or flow
- line attribution, where available, does not match the task intent

<Warning>
  Neroli is optimized for parallel work, not blind merging. Keep the parts you want.
</Warning>
