Neroli is most useful at review time. Treat every session output as a proposal
until you inspect the diff and verify the behavior.

## Review checklist

<Steps>
  <Step title="Open the diff">
    Inspect every changed file. Look for unrelated edits, generated files, and accidental secrets.
  </Step>

  <Step title="Read the session summary">
    Compare the plan, commit story, and agent summary with the actual diff.
  </Step>

  <Step title="Run verification">
    Use a saved Verify or Build action, browser checks, simulator checks, or a terminal command from the same session.
  </Step>

  <Step title="Decide what to keep">
    Continue the session, manually edit, commit, open a pull request, merge, or archive the session.
  </Step>
</Steps>

## Red flags

- Changes outside the requested area.
- Local-only files such as `.env`, certificates, or build output.
- Tests removed instead of fixed.
- New commands or scripts that are destructive by default.
- A passed check that was run from a different session.
- Conflict warnings where another session touched the same file or flow.
- Line attribution, where available, that does not match the intent of the task.

<Tip>
  If two sessions solve the same problem, review both diffs before deciding. The better branch may not be the one that finished first.
</Tip>
