Browser panes keep web verification attached to the session that owns the code
changes. That matters because the page, local server, console output, and network
evidence all need to match the branch you are reviewing.

## When to use a browser pane

- A visual change needs inspection.
- A bug only appears after clicking through a flow.
- The agent needs to read page state before editing.
- The review needs evidence beyond "the command passed."

## Recommended flow

<Steps>
  <Step title="Start the app from the session">
    Use the repository's Run action or terminal pane so the server belongs to the same branch you are reviewing.
  </Step>

  <Step title="Open the page in Neroli">
    Use the app UI for normal review. Agents can use the browser CLI when they need structured page state.
  </Step>

  <Step title="Inspect before changing">
    Confirm the page, visible controls, and relevant state before asking an agent to click or edit.
  </Step>

  <Step title="Capture the final evidence">
    Keep the final visible state, failure, or network result attached to the session review.
  </Step>
</Steps>

## Agent CLI examples

```sh
neroli browser open http://localhost:4321 --name website
neroli browser look --browser website
neroli browser tap "Save"
neroli browser wait-for-text "Saved"
```

<Warning>
  Agents should use the Neroli browser pane for this workspace. Separate browser tools cannot see the same page the user is reviewing.
</Warning>
